Controller, motor torque & inertia: where the physics lives — and where it doesn't

A plain-language map of the simulation's control loop and plant model, what each physical quantity corresponds to in code, and which of them you can actually turn. Sourced from turret_model.py, control_system.py, pi_controller.py, stewart_platform.py and app.py.

The short answer: the sim now ships two selectable plants. The default is a kinematic velocity servo: the controller commands an angular rate (deg/s) and the actual rate chases it through a single first-order lag, τ = 0.2 s — that one number stands in for the whole drivetrain (torque, inertia, friction, gearbox, current loop), and the sections below describe it. The second plant is a torque-driven second-order model where those quantities are explicit: inertia J, viscous damping B, a motor torque constant K_t with a current limit i_max (so torque saturates at τ_max = K_t·i_max), and Coulomb friction with a Karnopp stiction band; there the inner PI commands a current. A toggleable white sensor noise can also be injected on the controller's measurements. See control_theory.md §3b/§3c for the equations.

01 · Signal chain

One axis, end to end

Azimuth and elevation each run an identical, independent chain (a classic cascaded servo). The gold dashed box is where a torque model would sit — today it's collapsed into the τ lag.

What each stage does

Outer position loop — pure P (Ki = 0). Turns a pointing error (rad) into a speed reference (rad/s), clamped to ±180 deg/s. control_system.py:87

Inner speed loop — PI with back-calculation anti-windup. Its feedback is the absolute line-of-sight rate (gimbal rate + analytic base rate — what a rate gyro on the barrel would read), so base motion shows up as speed error and gets rejected. Output: a rate command u, clamped to ±180 deg/s. pi_controller.py:43

Plant — per axis, semi-implicit Euler at dt = 0.01 s:

u     = clip(u, ±rate_limit)          # ±180 deg/s
ω    += (u − ω) · dt / τ              # actual rate lags the command, τ = 0.2 s
φ    += ω · dt                        # angle integrates the actual rate

That's the whole plant (turret_model.py:55–65). No force, no torque, no mass — the "motor" is the assumption that the rate converges to the command with a 0.2 s time constant.

02 · The physics dictionary

Where torque and inertia went

Every real-drivetrain quantity is either lumped into τ, implied by it, or absent.

Inertia J → lumped into τ

In a real velocity-controlled servo, the closed rate loop behaves roughly first-order with a time constant set by inertia over drive stiffness (≈ J/b, or 1/bandwidth of the vendor's velocity loop). Here that whole story is the single number τ = 0.2 s. Raise τ ⇒ "heavier turret / weaker drive"; lower τ ⇒ stiffer, snappier axis. Both axes share one τ — a real turret's azimuth ring (which carries everything) would be slower than elevation.

Motor torque → implied, uncapped

Torque would be τ_m = J·ω̇. The model's acceleration is ω̇ = (u − ω)/τ, so the implied torque is proportional to rate error — and is never saturated. Worst case (command +180 deg/s while spinning −180) the axis accelerates at 1800 deg/s²; the sim happily "produces" whatever torque that takes. A torque-limited motor would cap ω̇ at τ_max/J and show slew-rate limiting the current model can't.

Max motor speed → the rate limit

RATE_LIMIT = 180 deg/s (app.py:45) clips the command, playing the role of the motor's no-load speed / drive velocity limit. The same value clamps the outer loop's speed reference, so the cascade can't ask the inner loop for more than the plant accepts (that's what keeps anti-windup honest).

Friction, gravity, backlash → absent

No viscous or Coulomb friction, no gravity imbalance pulling elevation down, no gear backlash, no structural resonance, no sensor noise (noise is on the README roadmap). The only nonlinearities are the two saturations (rate command, ±90° elevation hard stop) — and the hard stop simply zeroes the rate, turret_model.py:76.

04 · Real torque dynamics — now implemented

The upgrade path is one function ✓ done — TorqueTurretModel

This upgrade has since been implemented as TorqueTurretModel (selectable in the panel); the choice below was taken option-A: the inner PI's output becomes a current command, no third loop. Coulomb friction uses a Karnopp stiction band and torque saturates at τ_max = K_t·i_max. The historical note below describes the (still-default) kinematic plant and the design sketch that was followed.

Everything torque-shaped funnels through the plant's _advance_axis. Swapping the first-order rate lag for a torque-driven second-order axis looks like:

# per axis: J·ω̇ = τ_m − b·ω − τ_load
tau_m  = clip(torque_cmd, ±tau_max)      # motor torque saturation (NEW param)
omega += (tau_m − b*omega − tau_grav) / J * dt   # J, b, tau_grav (NEW params)
angle += omega * dt

Then the inner PI's output becomes a torque command instead of a rate command (or you keep the rate command and add a third, innermost torque/current loop — the standard industrial triple cascade: position → velocity → current). New tunable physics you'd gain: per-axis J (azimuth carries the whole gimbal, so J_az > J_el), τ_max (acceleration saturation and honest slew limiting), b (damping), and a gravity imbalance torque on elevation. The controller code (control/) wouldn't need structural changes — it already only sees errors and gains — though the gains would need retuning since the plant order changes.

Until then, remember the honest reading of results: the sim validates loop architecture and disturbance-rejection logic, not actuator sizing. Nothing in it will ever torque-saturate, overheat, or fail to accelerate.
